Git分支管理

分支

在版本回退的时候，我们知道每次提交，Git都会把他们串成一条时间线，而这条时间线我们就称为分支，而我们创建一个新的版本库的时候，就只有一条分支，这条分支叫做master主分支

创建分支

执行git checkout -b xxx，这条命令相当于git branch xxx（创建xxx分支）和git checkout xxx（切换xxx分支），即创建并切换分支，执行git branch会查看当前分支，同时也会查询所有分支

分支合并

当我们创建一个分支并切换到该分支，无论你做什么操作，提交之后，只是提交到你当前分支，你切回主分支，是不会有任何改变，需要将之前操作的数据让主分支也有，就需要进行分支合并，执行git merge xxx，该命令用于合并指定分支到当前分支，合并之后我们就可以看到，之前在另外一个分支操作的数据在主分支也有了，这种合并方式为快进模式，即，直接将master分支指向另一个分支，所以速度非常快

分支删除

当我们合并了分支之后，不需要这只分支了，我们可以执行git branch -d xxx，执行后查看一下分支就会发现，这个分支被删除了

分支合并冲突

当我们创建并切换到该分支，进行操作后，切换为主分支，同样进行操作，提交之后进行分支合并，此时合并会导致冲突，从而无法合并，使用git status查看工作区可以看到冲突，此时我们需要自己手动编辑文件保留我们所需要的数据，之后提交，就可以解决分支冲突，可以使用git log --graph --pretty=oneline --abbrev-commit命令看到分支合并图

分支管理策略

一般我们在未做出任何指定时，我们使用的是快进模式进行分支合并，这种方式在删除分支之后，会丢掉分支信息，我们可以在合并的时候加上--no-ff参数，即表示禁用快进模式，此时Git会在merge时生成一个新的commit，这样就可以在分支历史上看出分支信息

“储藏”分支

有的时候我们在工作时，突然领导下发一个紧急bug需要修复，我们第一个想法就是创建分支来修复他，但是目前手头工作并没有完成，没办法提交，此时Git还提供了git stash命令，让我们将当前工作储藏起来，等完成修复之后在恢复，使用git stash list查看我们储藏的工作，此时我们有两个办法：
（1）使用git stash apply恢复，但是恢复后stash内容不会删除，需要我们手动执行git stash drop进行删除
（2）使用git stash pop，恢复的同时将stash内容删除
ps:你可以多次使用stash，恢复的时候，先查看stash列表，指定性的恢复

强制删除分支

我们进行分支删除的时候，直接使用git branch -d xxx进行删除，但是这个在分支已经进行了合并的情况下，假设你的分支并没有进行合并，执行该命令，会弹出删除失败，提示你分支并没有合并，如果被删除，将丢失修改，此时我们又不需要此分支，我们就可以使用git branch -D xxx进行强制删除

分支推送

当我们代码需要推送到远程仓库时，可以使用git push 仓库名 分支名进行推送

标签管理

有的时候，我们需要对某个版本进行打包，但是对于commit id又过于复杂，不好诉说，此时就需要标签

创建标签

在你需要打标签的分支上执行git tag xxx就可以打上一个新的标签，默认是在最新提交的commit上打标签，我们可以使用git tag查看所有标签，还可以指定commit id打标签，执行git tag xxx commit id，也可以创建带有说明的标签,git tag -a xxx -m "说明" commit id

操作标签

推送某个标签到远程仓库git push 仓库名 标签名
推送全部标签到远程仓库git push 仓库名 --tags
删除未推送到远程的标签，即本地标签git tag -d 标签名
删除远程标签，首先删除本地标签git tag -d 标签名，然后删除远程标签git push 仓库名 :refs/tags/标签名
